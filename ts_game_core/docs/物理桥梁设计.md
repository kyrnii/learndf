# 物理桥梁设计

项目新增了 `src/physics` 目录，用来承接统一物理接口与不同后端实现。

当前目标不是立刻接入真实物理引擎，而是先把桥梁层结构稳定下来，让后续接 `JS` 和 `native` 两套实现时不影响上层游戏代码。

## 目录结构

- `src/physics/core`
  统一类型、后端接口、`PhysicsWorld`、`PhysicsBody`
- `src/physics/backends/js`
  JS 物理后端入口
- `src/physics/backends/native`
  原生物理后端入口
- `src/physics/backends/stub`
  当前可运行的内存占位后端
- `src/physics/game`
  游戏侧统一入口 `Physics`

## 当前分工

### core

这一层只定义抽象，不关心底层到底是 Bullet、Rapier 还是原生封装。

它负责：

- `IPhysicsBackend`
- `PhysicsWorld`
- `PhysicsBody`
- 统一的 `Vec3 / TransformLike / RaycastInput / PhysicsBodyDesc`

### backends

这一层负责适配具体实现。

当前先提供：

- `JsPhysicsBackend`
- `NativePhysicsBackend`
- `StubPhysicsBackend`

现在的 `JsPhysicsBackend` 和 `NativePhysicsBackend` 都继承自 `StubPhysicsBackend`，只是占位，后续可以分别替换成真实实现。

目前 `JsPhysicsBackend` 已经接入 Rapier，并采用异步初始化方式：

```ts
import { JsPhysicsBackend, Physics } from "../src/physics";

const backend = await JsPhysicsBackend.create();
const physics = new Physics(backend, {
    gravity: { x: 0, y: -9.81, z: 0 },
});
```

之所以使用异步初始化，是因为 Rapier 的 WASM 运行时需要先完成加载。

### game

这一层提供给游戏逻辑直接使用的统一入口：

- `Physics`
- `EntityPhysics`

`Physics` 的职责是：

- 创建和持有物理世界
- 创建和销毁刚体
- 注册和删除障碍物
- 做 `raycast / overlapSphere`
- 同步 `Transform` 和物理体
- 提供运动相关操作

`EntityPhysics` 的职责是：

- 作为 `Entity.physics` 挂在实体上的原生物理句柄
- 让物理能力的接入风格更接近源码里的 `inst.Physics`

## 为什么命名为 Physics

这里没有使用 `PhysicsBridge` 这个名字，而是直接叫 `Physics`。

原因是这层对游戏逻辑来说就是“物理系统入口”，不需要额外让上层理解它是不是桥梁。

桥梁属性保留在内部结构里：

- 游戏层只依赖 `Physics`
- `Physics` 内部依赖 `IPhysicsBackend`
- 不同平台替换不同 backend

## API 设计原则

当前 API 尽量贴近源码使用习惯，但不直接暴露底层引擎对象。

例如：

- 可以有 `addObstacle`
- 可以有 `moveKinematic`
- 可以有 `raycast`
- 可以有 `overlapSphere`

但不直接把 Bullet 或其他引擎的原生句柄泄漏给上层。

## 障碍物与世界/寻路的关系

后续建议把“障碍物”视为同时影响三层的对象：

1. 物理层
   作为刚体或碰撞体存在
2. 世界层
   作为可查询对象存在
3. 导航层
   作为寻路阻挡存在

因此后续推荐流程是：

- 放置一个建筑或障碍物
- `World` 注册实体
- `Physics` 注册静态 obstacle
- `Navigation` 视需要更新局部阻挡

## 下一步建议

接下来最适合继续完善的是：

1. 给 `World` 增加对 `Physics` 的可选引用
2. 新增 `PhysicsBodyComponent` / `ColliderComponent`
3. 让障碍物放置时自动调用 `physics.addObstacle`
4. 在 `Locomotor` 或未来 `Navigation` 中逐步接入 `raycast` 和阻挡判断

当前这一步的目标只是把桥梁层立住，不是立刻完成完整物理系统。

## 当前接入风格

当前推荐把物理视为实体级原生子系统，而不是 gameplay component。

也就是说，后续更推荐这种风格：

```ts
entity.physics
entity.brain
entity.sg
entity.world
```

而不是：

```ts
entity.getComponent(PhysicsBodyComponent)
```

这样更贴近源码里 `Transform / Physics / AnimState / SoundEmitter` 这类原生对象的组织方式。
