# 世界模块说明

`src/world` 现在已经从“单张轻量可走地图”演进成“世界 + 多地图实例”的结构。

当前世界层的目标是：

- 支持多个独立地图实例
- 支持不同地图拥有不同的格子尺寸和边界
- 让放置系统优先依赖地图和占用，而不是依赖物理
- 让 `physics` 成为可选增强层，而不是必选前置

## 当前结构

- `src/world/World.ts`
  负责实体注册、地图管理、多地图切换、世界级查询。
- `src/world/MapContext.ts`
  表示一张独立地图实例，内部组合地形、占用、导航和可选物理。
- `src/world/TerrainMap.ts`
  保存静态地形、边界、基础不可走格。
- `src/world/OccupancyMap.ts`
  保存动态矩形占用，是放置系统的主真相源。
- `src/world/NavigationMap.ts`
  由地形图和占用图派生而来，负责可走性、直线检测和轻量绕障。
- `src/world/MapLink.ts`
  用来描述地图之间的连接点，方便后续做门、楼梯、传送点。
- `src/world/WalkableMap.ts`
  兼容旧接口的包装类，内部实际使用新的 `MapContext` 结构。

## 分层原则

### 1. TerrainMap

负责回答：

- 这个区域天生能不能走
- 这个区域天生能不能放
- 当前地图的格子尺寸和边界是什么

### 2. OccupancyMap

负责回答：

- 这块区域现在有没有被建筑或物体占住
- 某个矩形 footprint 能不能放下
- 某个已放置对象占了哪些格子

### 3. NavigationMap

负责回答：

- 当前位置到目标位置是否有直线通路
- 是否需要临时绕一下
- 后续真正接入路径搜索时，应该走哪些格子

它不是主真相源，而是 `TerrainMap + OccupancyMap` 的派生导航视图。

### 4. Physics

物理是可选增强层。

- 放置/经营/模拟玩法可以不开 physics
- 战斗/碰撞类玩法可以开启 physics 增强真实感

放置合法性不应依赖 physics 判定。

## 多地图实例

`MapContext` 表示一张独立地图实例。每张地图都可以有：

- 不同的 `cellSize`
- 不同的边界
- 不同的地形和占用数据
- 自己的导航视图
- 可选的 `physics`

这让下面这些场景都能自然支持：

- 大世界和房间共存
- 房间与大世界使用不同格子尺寸
- 某些地图启用 physics，某些地图关闭 physics

## 当前放置建议

推荐的放置流程是：

1. 先把世界坐标转换成格子坐标
2. 通过 `TerrainMap` 判断地形是否允许
3. 通过 `OccupancyMap` 判断 footprint 是否可放
4. 写入 `OccupancyMap`
5. 通知 `NavigationMap` 对应区域失效
6. 如果启用了 physics，再同步创建静态 obstacle

## 当前导航能力

目前 `NavigationMap` 还是最小版，主要提供：

- `isWalkable(...)`
- `isPathWalkable(...)`
- `findNearbyWalkableOffset(...)`
- `findSimpleDetour(...)`
- `invalidateCell(...)`
- `invalidateRect(...)`

这已经够支撑：

- `Wander`
- `RunAway`
- `Panic`
- `Locomotor` 的最小绕障

后面如果要继续升级，可以再接真正的 A* 或分层路径系统。
