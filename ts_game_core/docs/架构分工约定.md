# 架构分工约定

这份文档用于约束 `ts_game_core` 里几层核心系统的职责边界，避免后续开发时把逻辑混在一起。

当前推荐原则：

`能放 SG 的先放 SG，Action 保持薄，Behavior 负责决策，Component 负责长期能力与数据。`

## 1. Action 负责什么

`Action` 的职责是描述一个“可被发起的动作意图”。

当前阶段，`Action` 只负责这些事情：

- 动作 `id`
- 动作生效距离 `distance`
- 进入哪个 SG 状态 `sgState`
- 到命中帧后执行哪个回调 `fn`

适合放进 `Action` 的内容：

- 攻击、拾取、使用、交互这种“一个动作对应一次状态切换和一次帧回调”的逻辑
- 玩家输入和 AI 最终都要统一发起的动作意图
- 纯动作定义，不带太多状态控制

不建议先放进 `Action` 的内容：

- 很重的生命周期管理
- 多段状态流转
- 复杂打断规则
- 大量运行时状态缓存

一句话：

`Action` 更像“命令描述”，不是复杂流程管理器。

## 2. StateGraph 负责什么

`StateGraph` 负责动作的表现时序和状态切换。

适合放进 `SG` 的内容：

- 前摇、后摇、硬直、受击、死亡
- 动画时间轴
- 命中帧、特效帧、音效帧
- 是否允许被打断
- 忙碌态、攻击态、受击态、死亡态
- 引导、蓄力、多段攻击、连段分支
- 某个动作在不同阶段要切哪些状态

凡是和“这个动作怎么演”有关的内容，优先放 `SG`。

比如这些场景优先用 `SG` 解决：

- 蓄力攻击
- 连续斩击
- 施法读条
- 引导中被打断
- 攻击过程中不同帧产生不同效果

一句话：

`SG` 更像“动作导演”。

## 3. Behavior 负责什么

`Behavior` 负责“为什么做这个动作”，也就是决策层。

适合放进 `Behavior` 的内容：

- 找目标
- 追击
- 保持距离
- 巡逻
- 回家
- 逃跑
- 满足条件后发起某个 `Action`

`Behavior` 应该尽量少关心动作细节，只需要决定：

- 当前要不要做
- 对谁做
- 什么时候切换目标
- 是追击、等待、逃跑还是交互

不建议放进 `Behavior` 的内容：

- 命中帧逻辑
- 具体动画推进
- 攻击动作内部的多段流程

一句话：

`Behavior` 更像“大脑”，不是“演员”。

## 4. Component 负责什么

`Component` 负责实体长期存在的能力、数据和规则。

适合放进 `Component` 的内容：

- `Combat`：攻击力、攻击间隔、目标、伤害结算
- `Health`：血量、受伤、死亡
- `Locomotor`：移动、接近目标、推送动作
- `Inventory`：背包数据
- `Transform`：位置、朝向

`Component` 应该提供稳定能力给上层调用，而不是主导复杂表现流程。

比如：

- `Combat` 负责“能不能打、打多少、结算什么”
- `SG` 负责“什么时候挥出去”
- `Behavior` 负责“为什么现在要打”

一句话：

`Component` 更像“器官和能力模块”。

## 5. 推荐分工方式

当你要实现一个新功能时，可以按下面顺序判断：

1. 这是“想做什么”还是“怎么演出来”？
2. 如果是决策，放 `Behavior`
3. 如果是状态和时间轴，放 `SG`
4. 如果是长期能力和数据，放 `Component`
5. 如果只是一个统一发起入口，放 `Action`

## 6. 典型例子

### 例子一：普通攻击

- `Behavior`：发现敌人，决定攻击
- `Action`：定义 `ATTACK`
- `Locomotor`：接近到攻击距离
- `SG`：进入 `attack` 状态并在命中帧触发
- `Combat`：真正结算伤害

### 例子二：拾取物品

- `Behavior` 或输入层：决定去捡
- `Action`：定义 `PICKUP`
- `Locomotor`：走到物品旁边
- `SG`：进入 `pickup` 状态
- `Inventory` / `Item`：完成拾取结算

### 例子三：蓄力技能

- `Behavior` 或输入层：决定释放技能
- `Action`：只负责发起技能动作
- `SG`：处理蓄力、释放、取消、中断
- `Component`：处理资源消耗、伤害、冷却数据

这个例子里，复杂度主要应该放在 `SG`，而不是把 `Action` 做厚。

## 7. 当前项目的建议

现阶段推荐保持：

- `Action` 轻量
- `SG` 负责复杂动作流程
- `Behavior` 先补齐基础行为节点
- `Component` 继续补核心能力

等后面出现这些情况时，再考虑把 `Action` 扩厚：

- 很多动作都在重复相同的发起前校验
- 需要统一输入队列
- 需要网络动作确认
- 需要回放和日志埋点
- 需要复杂动作生命周期管理

在那之前，优先把 `SG` 和 `Behavior` 的边界打稳，会更适合当前阶段。
